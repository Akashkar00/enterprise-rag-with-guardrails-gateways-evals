The Build, Step by Step

From zero to a live cloud deployment

This is the exact order I set things up. Each step builds on the last — you can follow it top to bottom to recreate the whole deployment.

1

Log in & point at my project gcloud auth

First I authenticated the CLI (twice — once for me, once for local Python scripts via Application Default Credentials) and set the active project so every later command targets it.

# Log in (browser) + local app credentials
gcloud auth login
gcloud auth application-default login

# Lock every future command to this project
gcloud config set project advanced-rag-project-500909
2

Turn on the Google Cloud APIs I need services enable

Nothing works until the APIs are switched on. I enabled everything the system touches in one shot — containers, AI, storage, database, and the event system.

gcloud services enable \
    artifactregistry.googleapis.com # image registry \
    run.googleapis.com # Cloud Run \
    cloudbuild.googleapis.com # CI/CD builds \
    sqladmin.googleapis.com # Cloud SQL \
    documentai.googleapis.com # PDF parsing \
    aiplatform.googleapis.com # Vertex AI embeddings \
    redis.googleapis.com eventarc.googleapis.com pubsub.googleapis.com \
    compute.googleapis.com
3

Create the storage buckets Cloud Storage

Two buckets: one where raw documents get uploaded, and one where the parsed, cleaned text is saved. The raw bucket also becomes the trigger source for automatic ingestion later.

gcloud storage buckets create gs://advanced-rag-project-500909-rag-raw --location=us-central1
gcloud storage buckets create gs://advanced-rag-project-500909-rag-processed --location=us-central1
4

Grant permissions with IAM least privilege

Every actor gets only the roles it needs. My user account gets admin roles to build and deploy; the Compute service account (which actually runs the builds and services) gets runtime roles; and the Cloud Run service agent gets VPC access. A snapshot:

# Give the runtime service account access to AI, docs & storage
gcloud projects add-iam-policy-binding advanced-rag-project-500909 \
    --member="serviceAccount:872202826024-compute@developer.gserviceaccount.com" \
    --role="roles/aiplatform.user"          # Vertex AI embeddings
# repeated for: documentai.apiUser, storage.admin,
# artifactregistry.writer, logging.logWriter, vpcaccess.user …

Full role map is in the table further down. ⬇️

5

Create the image registry Artifact Registry

A private Docker repository to hold the container images. Cloud Build pushes here, Cloud Run pulls from here.

gcloud artifacts repositories create rag-repo \
    --repository-format=docker \
    --location=us-central1 \
    --description="Docker repository for RAG API"
6

Build all 4 images in the cloud Cloud Build

Instead of building Docker images on my laptop, Cloud Build does it in Google's cloud and pushes straight to Artifact Registry. My cloudbuild.yaml defines 4 parallel builds — one per microservice.

  • backend — FastAPI + LangGraph agent
  • ui — Streamlit chat interface
  • ingestion — document parser + embedder
  • evals — RAGAS evaluation dashboard
# Builds & pushes all 4 images from one config file
gcloud builds submit --config cloudbuild.yaml --timeout=7200 .
7

Deploy the containers to Cloud Run serverless

Each image becomes a live, auto-scaling service. Config (API keys, bucket names, project ID) is injected as environment variables at deploy time, so no secrets live inside the image.

gcloud run deploy rag-api \
  --image us-central1-docker.pkg.dev/advanced-rag-project-500909/rag-repo/rag-api:v1 \
  --region us-central1  --memory 2Gi  --timeout=300 \
  --set-env-vars "PROJECT_ID=advanced-rag-project-500909" \
  --set-env-vars "LOCATION=us-central1" # … + AI, bucket, DB vars

Four services go live: rag-api, rag-ui, rag-ingestion, rag-evals — all IAM-authenticated.

8

Add persistent memory Cloud SQL · Postgres 15

The agent needs to remember conversations across restarts and scale-to-zero. Cloud SQL Postgres 15 stores that memory. Cloud Run connects over a secure unix socket (/cloudsql/…) — no public IP. If the DB is ever unreachable, the code automatically falls back to in-memory storage so nothing crashes.

  • Database: rag_memory — written by LangGraph's PostgresSaver
  • Connection: unix socket, authenticated with the cloudsql.client IAM role
9

Add a semantic cache + private networking Memorystore · VPC

Memorystore Redis holds recent answers. When a new question is semantically close to an old one (cosine distance < 0.15), the cached answer returns in ~50 ms — skipping the whole LLM pipeline. Redis lives on a private IP, so Cloud Run reaches it through Direct VPC Egress on rag-vpc rather than the public internet.

  • Redis Memorystore — semantic cache, private IP
  • Direct VPC Egress — Cloud Run → Redis + Cloud SQL over internal network
10

Wire up automatic ingestion Eventarc · Pub/Sub

The final piece makes ingestion hands-free. An Eventarc trigger listens for the object.finalized event on the raw bucket. The moment any file is uploaded, it POSTs to the ingestion Cloud Run service, which parses, chunks, embeds, and indexes it — fully automatic, verified end-to-end.

# Test it: just drop a file in the bucket…
gcloud storage cp any_doc.pdf gs://advanced-rag-project-500909-rag-raw/
# …and watch ingestion fire on its own
gcloud run services logs read enterprise-rag-ingestion --region us-central1

Going Further

Scaling it with Infrastructure-as-Code

Beyond the manual gcloud path, I also captured the whole stack in Terraform, so the entire environment — VPC, Redis, Cloud SQL, all 4 Cloud Run services, and the Eventarc trigger — can be created or destroyed with a single command.

# 1 — create the image repo first
terraform apply -target=google_artifact_registry_repository.repo
# 2 — build & push all images
gcloud builds submit --config cloudbuild.yaml --timeout=7200 .
# 3 — provision everything else (VPC, Redis, SQL, Cloud Run, Eventarc)
terraform apply
In One Sentence

The whole thing, summarized

I enabled the APIs, created buckets + IAM roles, built my containers with Cloud Build into Artifact Registry, deployed them on Cloud Run, gave the agent persistent memory (Cloud SQL) and a fast semantic cache (Memorystore Redis) over a private VPC, and used Document AI + Vertex AI to read and understand documents — all triggered automatically by Eventarc the moment a file is uploaded.
